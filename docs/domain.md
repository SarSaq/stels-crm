# Stels CRM — Доменная модель и источник правды

> Живой документ. Описывает предметную область, производственный цикл, роли и принятые
> решения. От него строится webapp. Правь свободно — при расхождении кода и этого файла
> считаем правдой то, что здесь согласовано.

Дата последнего обновления: 2026-07-06

---

## 1. Что это за система

Производственная CRM для рекламно-производственной компания (широкоформатная печать).
Управляет заказами от приёма менеджером до отгрузки готовой продукции.

**Цель проекта:** заменить текущую Google-таблицу «Заказы 2026» современным webapp
(деплой на Vercel), сняв ограничения таблиц: конкурентная работа цеха, скорость,
разграничение прав, история этапов, аналитика.

**Важно:** текущая живая таблица НЕ трогается — в ней работают люди. Новая система строится
параллельно. История переносится из выгруженного слепка (CSV), а не из живого листа.

---

## 2. Ключевые сущности

- **Заказ** — центральная сущность. ~6700 реальных заказов в текущей таблице.
- **Этап производства** (цех) — операция над заказом со своим статусом. У заказа их несколько.
- **Справочники** (лист «Параметры») — списки для выбора: менеджеры, исполнители, материалы,
  способы печати, оборудование, клиенты, статусы и т.д.

---

## 3. Производственный цикл — машина состояний (статус заказа, поле J)

Статус заказа — **двунаправленный**: его меняют вручную роли, но он же автоматически
пересчитывается от статусов этапов.

### Основные статусы заказа

| Статус | Кто ставит | Смысл |
|---|---|---|
| **Ожидание** | Менеджер | «Бронь» строки за менеджером + заполнение полей. *(в webapp → честная стадия «Черновик» + явный владелец)* |
| **Проверка** | Препресс | Проверка и подготовка файлов к печати |
| **В работе** | Препресс (вручную) / авто | Запуск производства; авто-старт всех активных этапов |
| **В процессе** | авто | Хотя бы один этап в работе/выполнен, но не все |
| **ОТК** | авто | Все активные этапы «Выполнено» → контроль качества и упаковка |
| **Готово** | ОТК | Проверено и упаковано, готово к отгрузке |
| **Отгружено** | ОТК | Передано в транспорт |
| **Отмена** | Менеджер | Заказ отменён |

*(В справочнике «Параметры» есть ещё статусы: Постпресс, Инженер, Монтаж, Макетка —
используются вручную/редко, не участвуют в авто-логике скрипта. Уточнить надобность.)*

### Переходы

```
Черновик/Ожидание ─(менеджер)→ Проверка ─(препресс)→ В работе
     │                                                    │
     │                                          (авто, от этапов)
   Отмена                                                 ▼
                                          В процессе → ОТК → Готово → Отгружено
                                                              (ОТК)   (ОТК)
```

### Правила авто-логики (из Apps Script — переносим в бэкенд)

**Сверху вниз (изменён статус заказа):**
- `→ В работе`: всем активным этапам (где стоит галочка/выбран материал) с пустым статусом
  проставить «В работе».
- `→ Готово` / `→ Отгружено`: все активные этапы принудительно «Выполнено».

**Снизу вверх (изменён статус этапа):**
- Все активные этапы «Выполнено» → заказ = **ОТК**.
- Хотя бы один «В процессе» или «Выполнено» → заказ = **В процессе**.
- Хотя бы один «В работе» → заказ = **В работе**.

**Защиты (обязательно сохранить):**
- Если статус заказа пустой / «Ожидание» / «Готово» / «Отгружено» — логика «снизу вверх»
  НЕ срабатывает (защита от отката финальных статусов).
- «ОТК» не откатывается назад в «В процессе».
- Автодата: при первом выставлении статуса, если «Дата создания» пуста — проставить текущую.
  *(В webapp даты храним как настоящие timestamp, формат «06 июл» — только для отображения.)*

## 4. Этапы производства (цеха)

У этапа два аспекта: **активен ли** (нужен для этого заказа) и **статус выполнения**.

Статусы этапа: `пусто → В работе → В процессе → Выполнено`.

### Активные этапы (6)

| Этап | Активен, если | Примечание |
|---|---|---|
| Печать | стоит галочка «Печать» | |
| Проварка и люверсы | галочка «Проварка и люверсы» | |
| Ламинация | выбран материал ламинации (текст, не галочка) | тип: Мат/Глянец |
| Резка | галочка «Плоттер» ИЛИ «Каттер» | два входа, один статус |
| Фрезер | галочка «Фрезер» | |
| Сварка | галочка «Сварка» | |
| Макетка | галочка «Макетка» | ⚠ см. баги ниже |

### Информационные флаги (НЕ этапы, без статуса)

«Нанесение скотча», «Выборка», «Монтажная пленка» — просто пометки на заказе.

## 5. Роли пользователей (5)

| Роль | Где работает | Устройство | Права |
|---|---|---|---|
| **Менеджер** | где угодно | десктоп + телефон | Создаёт заказ; статус: Черновик/Проверка/Отмена; организует отгрузку |
| **Препресс** | **удалённо** | десктоп | Готовит файлы: превью, ссылка на печать, название файла; статус → «В работе» |
| **Производство** (печатники/исполнители) | цех | терминал/планшет | Меняют статус ТОЛЬКО своего этапа; статус заказа (J) не трогают |
| **ОТК** | **только в цеху** | терминал/планшет | Статус: ОТК → Готово → Отгружено; контроль качества и упаковка |
| **Руководитель** | где угодно | десктоп | Полный обзор + аналитика |

**MVP:** видимость НЕ ограничиваем — все видят все заказы. Роли закладываем в модель данных
сразу, чтобы позже включить раздельные интерфейсы по узлам производства без переделки.

---

## 6. Баги текущей таблицы — НЕ переносим

В новой системе делаем чистую модель, эти дефекты исправляем:

1. **Макетка**: её статус пишется в колонку «Сварочный цех», а не в свою — рассинхрон колонок.
2. Сдвиг/путаница парных колонок «этап/статус» в правой части листа.
3. «Замеры» — этапа нет, лишний (в скрипте фигурировал, но колонки не существует). Убран.
4. Даты в свободном тексте («11 янв.») — переходим на настоящие timestamp.

## 7. Принятые технические решения

**Архитектура — три слоя, ФИНАЛ:**

```
┌─────────────┐     ┌──────────────────────┐     ┌─────────────────────┐
│  Сотрудник  │ ──► │   VERCEL             │ ──► │  SUPABASE            │
│  (браузер)  │     │  • Интерфейс (UI)    │     │  • Postgres (заказы) │
└─────────────┘     │  • Бэкенд-логика     │     │  • Storage (превью)  │
                    │    (машина состояний)│     │  • Auth (логины)     │
                    └──────────────────────┘     └─────────────────────┘
```

- **Хранилище данных:** **Supabase (PostgreSQL)** — управляемый Postgres с табличным
  веб-интерфейсом. Даёт настоящую БД «на вырост» И привычный табличный вид для ручного
  просмотра/правки. У заказчика уже есть опыт в Supabase. Требований хранить данные в РФ нет
  (данные пользователей не хранятся, это внутренний сервис).
- **Бэкенд/логика:** **на Vercel** (serverless-функции). Вся машина состояний из Apps Script
  переезжает сюда — единый «мозг» в одном месте (уходим от размазанной логики).
- **Хостинг/интерфейс:** **Vercel**.
- **Авторизация:** Supabase Auth (логины для ролей менеджер/препресс/производство/ОТК/руководитель).
- **Хранилище картинок (превью):** **Supabase Storage**. Превью лёгкие (20–30 КБ), объём мизерный
  (~170 МБ на 6700 заказов — влезает в бесплатный 1 ГБ надолго).
  **Уходим от Cloudinary** (нестабилен в РФ). Старые ссылки Cloudinary в истории оставляем как есть.
- **Трафик картинок:** сотрудники активно листают → ОБЯЗАТЕЛЬНО кэш браузера + lazy-load +
  мелкие миниатюры в списке. Повторные загрузки не тратят трафик → реальный расход ~3% лимита Free.
- **Хранилище картинок — сменный модуль:** если Supabase Storage закапризничает из РФ или
  объёмы вырастут — переключаемся на **Yandex Object Storage (S3)** без переписывания логики.
  (Именно Object Storage/S3, НЕ Яндекс Диск: у Диска нет прямых ссылок и есть лимиты API — для
  раздачи файлов приложению не годится.)
- **Загрузка превью:** основной способ — кнопкой из карточки заказа (надёжная авто-привязка
  к заказу); опционально — «папка синхронизации» (файл с номером задачи) как привычный вход.
- **Исходники макетов (тяжёлые PDF/TIFF):** НЕ в облако. Остаются на файловом сервере
  (`Z:\`, `V:\`), в CRM — только ссылка, как сейчас.
- **Расчёт цены заказов:** ВНЕ скоупа. Система только про производство, без денег.
- **Импорт истории:** из выгруженных `orders.csv` / `params.csv`.

## 8. Дорожная карта улучшений (сверх Sheets)

Приоритет для MVP отмечен ⭐.

1. ⭐ Явная «бронь»/владелец заказа вместо статуса «Ожидание» → статус отражает реальную стадию.
2. ⭐ Канбан-доска по статусу заказа (перетаскивание = смена статуса с проверкой прав).
3. ⭐ Этапы как объекты с историей (кто/когда/какой переход) → аналитика узких мест.
4. ⭐ Строгие права в бэкенде (печатник не может тронуть статус заказа и т.п.).
5. ⭐ Загрузка превью из карточки заказа.
6. Очередь «мои задачи» для рабочего (будущие раздельные интерфейсы по цехам).
7. Уведомления (заказ дошёл до ОТК / готов к отгрузке).
8. Авто-расчёт площади печати (ширина × высота × кол-во) и нормальные даты.
9. Шаблоны / «повторить заказ» (много повторяющихся клиентов, напр. ДНС — 443 заказа).
10. Быстрый поиск и фильтры по тысячам заказов.

## 10. Структура текущих данных (справочно)

`# · Заказчик · Добавил(менеджер) · Дата создания · Дата сдачи · Приоритет · Превью ·
Описание · Ссылка на исходник · Статус(J) · Кол-во · Ширина,мм · Высота,мм · Пл.печати м² ·
Материал · Материал в наличии · Способ печати · Кол-во сторон · Цветность · Назв-е файла ·
Ссылка(печать,U) · Ссылка превью` + правый блок парных колонок этапов.

Лист «Параметры»: справочники-словари (менеджеры, исполнители, материалы, способы печати,
оборудование, статусы, цвет, срочность, стороны, клиенты, статусы задач).
