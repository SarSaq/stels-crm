## Главная идея (в двух словах)

В Google-таблице всё было на одном листе в 43 колонки, где этапы производства — это
парные колонки-«костыли». В нормальной БД мы раскладываем это на **несколько связанных
таблиц**, каждая про одну вещь. Это и убирает баги, и даёт скорость, права, историю.

Таблиц будет немного — 8 штук:

| Таблица | Про что | Аналог в старой системе |
|---|---|---|
| `orders` | Заказы (карточка) | строки листа «Заказы 2026» |
| `order_stages` | Этапы производства заказа | правый блок парных колонок |
| `stage_types` | Справочник типов этапов (Печать, Сварка…) | «зашито» в скрипте |
| `clients` | Клиенты | столбец «Клиенты» в Параметрах |
| `materials` | Материалы | столбец «материалы» |
| `users` | Сотрудники + роли | «менеджеры»/«Исполнитель» + логины |
| `order_history` | История изменений (кто/что/когда) | ❌ не было, новое |
| `reference_values` | Мелкие справочники (способ печати, цвет…) | столбцы Параметров |

## 1. `orders` — заказы (сердце системы)

Одна строка = один заказ. То, что менеджер и препресс заполняют в карточке.

| Поле | Тип | Смысл |
|---|---|---|
| `id` | номер (авто) | Внутренний ID (=«#» из таблицы) |
| `client_id` | ссылка → clients | Заказчик |
| `manager_id` | ссылка → users | Кто добавил (менеджер) |
| `owner_id` | ссылка → users | **Владелец/«бронь»** (замена статусу «Ожидание») |
| `status` | статус | Статус заказа J (см. список ниже) |
| `description` | текст | Описание |
| `source_link` | текст | Ссылка на исходник (`Z:\...`) |
| `quantity` | число | Кол-во |
| `width_mm` | число | Ширина, мм |
| `height_mm` | число | Высота, мм |
| `print_area_m2` | число (авто) | Площадь печати = Ш×В×кол-во (считается сама) |
| `material_id` | ссылка → materials | Материал |
| `material_available` | да/нет | Наличие материала на складе (Да/Нет). Логика «Фрез.» → в очерёдность этапов |
| `print_method` | текст | Способ печати (УФ/Сольвент/Без печати) |
| `print_equipment` | текст | Оборудование (УФ планшет, Сольвент 1,6…) |
| `sides` | число | Кол-во сторон (0/1/2) |
| `color` | текст | Цветность (CMYK / CMYK+W / Белила) |
| `file_name` | текст | Назв-е файла (колонка T) |
| `print_link` | текст | Ссылка для печати (колонка U, `V:\...`) |
| `preview_url` | текст | Ссылка на превью (картинка) |
| `is_urgent` | да/нет | **«Срочно»** — ставит менеджер |
| `is_fire` | да/нет | **«🔥 Огонь»** — ставит руководитель произв.: сигнал «очень срочно» |
| `due_date` | дата | Дата сдачи |
| `flag_scotch` | да/нет | Пометка «Нанесение скотча» |
| `flag_vyborka` | да/нет | Пометка «Выборка» |
| `flag_mount_film` | да/нет | Пометка «Монтажная пленка» |
| `tech_notes` | текст | Технические характеристики печати |
| `created_at` | дата+время | Дата создания (настоящий timestamp) |
| `updated_at` | дата+время | Последнее изменение |

**Статусы заказа (`status`)** — назначение каждого (уточнено заказчиком):

Основная линия (авто-логика):
`Ожидание → Проверка → В работе → В процессе → ОТК → Готово → Отгружено` + `Отмена`

Специальные (ручные, ветвления):
| Статус | Кто ставит | Смысл |
|---|---|---|
| **Постпресс** | авто/цех | Печать готова, но нужна послепечатная обработка (резка, ламинация, люверсы). Стадия между печатью и завершением этапов |
| **Инженер** | препресс | Нужно вмешательство инженера (сложные схемы/расчёты). Инженер доделывает → возвращает в «В работе» |
| **Монтаж** | ОТК/менеджер | После «Готово», если НЕ отгружаем, а монтажники выезжают устанавливать изделие |
| **Макетка** | — | Этап цеха (см. ниже) — не нужен как отдельный статус заказа |

> **Постпресс** — часть основной линии: означает «напечатано, идёт постобработка».
> **Инженер** работает между «Проверка» и «В работе».
> **Монтаж** — альтернатива «Отгружено» после «Готово».

## 2. `order_stages` — этапы производства заказа

Тут исправляется главный «костыль» старой таблицы. Вместо парных колонок —
**отдельная строка на каждый этап заказа**. У заказа столько строк, сколько у него этапов.

| Поле | Тип | Смысл |
|---|---|---|
| `id` | номер (авто) | ID строки этапа |
| `order_id` | ссылка → orders | К какому заказу относится |
| `stage_type_id` | ссылка → stage_types | Какой это этап (Печать/Сварка…) |
| `is_active` | да/нет | Нужен ли этап для этого заказа (старая «галочка») |
| `sequence` | число | **Порядок выполнения** — можно перетаскивать (см. ниже) |
| `status` | текст | Статус этапа: пусто / В работе / В процессе / Выполнено |
| `detail` | текст | Доп. значение (напр. для Ламинации — Мат/Глянец) |
| `assignee_id` | ссылка → users | Кто выполняет (задел на будущее, п.5) |
| `started_at` | дата+время | Когда взят в работу |
| `done_at` | дата+время | Когда завершён |

Пример: заказ №2099 требует Печать + Ламинацию + Резку → у него **3 строки** в этой таблице.
Больше никакой путаницы «Макетка пишется в колонку Сварочный цех».

### 🔑 Очередность этапов (`sequence`) — важная новая фича

Из-за случая «Фрез» (материал сначала режут на фрезере, потом печатают) стало ясно:
**порядок операций у заказа не фиксирован** — иногда резка до печати, иногда после.

Решение: у каждого этапа заказа есть номер порядка (`sequence`). Менеджер или руководитель
производства **перетаскивает этапы местами** в карточке заказа → печатник/цех видит
правильную последовательность и понимает, что делать раньше.

Это заменяет и оптимизирует нынешнюю пометку «Фрез.» в поле «Материал в наличии»:
теперь «резать до печати» = этап «Резка/Фрезер» стоит в очереди **перед** «Печать».

*(«Материал в наличии» остаётся отдельным полем со значениями Да/Нет — только про наличие
на складе. Логика «сначала фрезер» переезжает в очерёдность этапов.)*

## 6. `users` — сотрудники и роли

| Поле | Смысл |
|---|---|
| `id` | ID (связан с Supabase Auth — логином) |
| `full_name` | Имя (Аня Рудак, Саргис…) |
| `role` | Роль: manager / prepress / production / qc / director |
| `is_active` | Работает ли сейчас |

**Роли:** менеджер (manager), препресс (prepress), производство (production), ОТК (qc),
руководитель (director), **инженер (engineer)**, **монтажник (installer)**, **макетчик (assembler)**.
Права каждой роли — из `docs/domain.md`, раздел 5. Проверяются в бэкенде.

> **«Исполнитель» = препресс + сотрудники цеха** (это не отдельные люди, а роль исполнителя
> у этапов). Поэтому отдельной таблицы «исполнители» НЕ делаем — все в `users` с ролями.
> Запись «кто именно сменил статус этапа» (`assignee_id`) — задел на будущее (п.5), в MVP не заполняем.
>
> **Макетка** — сотрудники цеха другого отдела (клеят/собирают объёмные буквы, таблички) →
> это роль/этап, а не статус заказа.
> **Инженер** — подключается по статусу «Инженер». **Монтажник** — по статусу «Монтаж».

## Как это соединено (связи)

```
clients ──┐
users ────┼──► orders ──► order_stages ──► stage_types
materials ┘        │
                   ├──► order_history
                   └──► (preview_url → Supabase Storage)
```

Читается так: «у заказа есть клиент, менеджер, материал; у заказа много этапов;
каждый этап — определённого типа; все изменения пишутся в историю».

---

## Что это нам даёт (чего не было в Sheets)

1. **Нет багов парных колонок** — этап это строка, а не «колонка +1».
2. **Скорость** — поиск по заказу/клиенту мгновенный даже на десятках тысяч строк.
3. **Права** — на уровне БД: печатник не может изменить `orders.status`.
4. **История** — кто/что/когда, и аналитика из неё.
5. **Целостность** — нельзя выбрать несуществующего клиента/материал.
6. **Авто-поля** — площадь печати и даты считаются сами.
